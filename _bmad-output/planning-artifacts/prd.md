## Executive Summary / 执行摘要

KInfoGit 目前是一套基于 Next.js 14 静态导出的个人网站与作品集。项目已经搭建好基础骨架与技术栈，但 About、Work 以及若干二级页面和功能尚未完整实现，对外展示更像是「搭好了框架的半成品」。本次迭代的目标，是将网站推进到一个可对外使用的 1.0 版本：在不引入不必要复杂度的前提下，补完核心页面与信息架构，让现有内容能够稳定、完整、体面地呈现。

在这个基础之上，本次迭代会刻意关注内容的结构化程度和信息组织方式。我们希望访问者能够在简单直观的导航与分组下，快速找到自己关心的项目或内容，同时在页面布局与信息层级中，自然感受到站点所有者在工程实践和产品思维上的体系化能力。

### What Makes This Special / 特别之处

与很多只追求视觉效果的个人作品集不同，KInfoGit 的这次迭代更强调「结构化信息」与「工程化的内容组织」：

- 页面与内容结构会围绕项目和经验的真实使用场景来设计，而不是简单堆砌模块；
- 项目信息会以更统一、可扩展的方式建模，便于后续按标签、类型或时间线进行筛选和定位；
- 导航和布局会刻意呈现从「整体概览 → 代表性项目 → 具体细节」的层次，让读者在很短时间内看出你的工程/产品体系化思考方式，而不仅是单点的技能罗列。

本次迭代不会一次性解决所有内容管理自动化的问题，但会为后续的项目结构化管理与潜在工具化改造打下基础：通过清晰的信息架构和项目字段设计，为未来的脚本化、配置化甚至接入 CMS 留出空间。

## Project Classification / 项目分类

**Technical Type:** web_app  
**Domain:** general（通用软件 / 个人网站）  
**Complexity:** low（领域复杂度），本轮实现为中等复杂度的信息架构与体验增强  
**Project Context:** Brownfield – 在既有 Next.js SSG 个人站基础上补完与扩展

结合现有文档与对话，这次工作聚焦在一个已有代码与页面结构的个人网站之上补齐核心页面和二级信息架构，确保基于当前内容即可对外发布 1.0 版本。同时，通过更有秩序的页面分层、项目信息结构与导航设计，让访问者既能快速检索到自己感兴趣的项目，又能从整体体验中感受到站点所有者在工程实现和产品设计上的系统性。

## Success Criteria

### User Success

- HR/招聘方在进入站点 10–20 秒内即可看到代表项目列表和你的基本背景，通过 1–2 次点击（首页→列表→详情）能深入至少 1–2 个项目的详细拆解。
- 访问者能轻松找到文章/观点入口，对你的思考方式和方法论形成直观印象。
- 访问结束后，HR/面试官对「是否推进你进入下一步」有清晰倾向，而不是「看完还是摸不清你是干什么的」。

### Business Success

- 在站点 1.0 上线后的 3–12 个月内，你在大部分投递或机会沟通中愿意主动附上该站点链接。
- 有一定比例的 HR/面试官在沟通或面试中主动提及站点内容（例如「我看了你的网站，那个项目……」）。
- 通过该站点实际促成若干次面试或合作机会（不追求精确计数，但你能明显感觉到「是网站帮忙」）。

### Technical Success

- 导航与路由结构清晰，核心访问路径上无明显 404 或断链问题。
- 首页、项目列表页和项目详情页在桌面与移动端均具有良好性能表现，不出现明显卡顿或异常缓慢加载。
- 新增或修改一个项目时，所需改动的文件和步骤是可控的，不需要在多处复制粘贴，同一信息尽量只维护一份权威来源。

## Product Scope

### MVP – Minimum Viable Product（网站 1.0）

- 补全核心页面与信息架构：至少包含首页、About、Work（或等价的项目入口）以及必要的二级页面，使基于当前已有内容即可完整对外展示。
- 在首页或等价入口中，直接呈现代表项目列表，并提供清晰的导航到项目详情页面。
- 为文章/观点提供稳定且清晰的入口，即使当前文章数量有限，也能支撑后续逐步补充。
- 保证基础的稳定性与性能：无明显 404/断链，核心页面加载体验良好。

### Growth – Post-MVP 增长特性

- 丰富项目结构化信息（标签、类型、时间线等），支持更加灵活的筛选与浏览体验。
- 对文章/观点区域进行扩展与分类，使不同类型内容（技术、产品、随笔等）更易被发现。

### Vision – Longer-term 方向

- 通过更结构化的数据模型和信息架构，为未来接入脚本化维护、轻量 CMS 或多语言扩展打下基础。
- 将 KInfoGit 从「个人作品集网站」演进为一套可持续演化、工程化程度更高的个人数字档案。

## User Journeys

### Journey 1: HR / 招聘专员 – 快速判断是否值得推进

- **入口来源**  
  HR 在筛选简历或候选人资料时，通过：简历 / 求职系统中的个人网站链接，或你在 LinkedIn / 脉脉 个人资料中留下的站点链接，点进你的个人网站。
- **第一印象与首页动作**  
  打开首页后的前 10–20 秒内，HR 看到一段简短清晰的个人定位，以及若干代表性项目的卡片或列表入口，不需要思考导航结构就能意识到「这里有我要看的项目」。
- **浏览路径**  
  典型路径是：先在首页快速扫个人定位和代表项目标题 → 进入 Work/Projects 或等价入口查看完整项目列表 → 点开 1–2 个项目详情页，重点看项目类型、你的角色和使用的主要技术/能力。
- **结束时的判断瞬间**  
  看完 1–2 个项目后，HR 结合岗位需求，很快得出「可以安排下一步」或「暂时不太匹配」的判断。成功的情况是：站点让这个判断过程顺畅、轻松，而不是增加困惑。

### Journey 2: 面试官 / 技术负责人 – 面试前做深入了解

- **入口来源**  
  面试官通常在面试前从 HR 处拿到你的站点链接，或在与你约面试、沟通时收到你主动分享的链接，然后在正式面试前抽几分钟打开浏览。
- **关注重点**  
  进入网站后，面试官主要想确认：你做过哪些有一定规模或复杂度的项目；你在这些项目中承担了什么职责；你是否展现了对工程和产品的系统性思考，而非仅是功能堆砌。
- **典型浏览路径**  
  理想路径是：先在首页扫一眼角色定位和代表项目列表，判断方向是否对口 → 选择 1–2 个与团队技术栈或业务相近的项目进入详情 → 在项目详情中重点阅读背景与目标、你的具体职责、关键技术/产品决策及其原因 → 如有余力，再进入文章/观点入口，查看是否有与架构、产品设计或工程实践相关的内容以加深印象。
- **决定“值得聊聊”的瞬间**  
  当面试官在 1–2 个项目详情中看到问题、方案、取舍与结果被成体系地讲清楚，并能看出你不是简单罗列技术名词，而是围绕明确目标做出判断时，会认为「这个人值得花时间好好聊聊」，带着更有针对性的问题进入面试现场，而不是从零开始。

### Journey Requirements Summary

- 首页需要同时呈现清晰的个人定位与明显可见的代表项目入口，支持在 10–20 秒内让不同角色理解「你是谁、做什么」。
- 需要一页结构清晰的项目列表页（如 Work/Projects），帮助 HR 和面试官快速扫描你的代表项目集合。
- 每个项目需要有结构化的详情页，稳定呈现背景、目标、你的角色、关键决策与结果等关键信息，避免只剩技术名词堆砌。
  - 需要稳定且可发现的文章/观点入口，即使当前文章数量有限，也能让有兴趣的面试官找到与你思考方式相关的内容。
  - 整体导航需简单直观，从任意入口（简历、LinkedIn、脉脉）点进来，在极少点击内即可完成上述浏览路径。

## Web App Specific Requirements

### Project-Type Overview

KInfoGit 是一个以内容展示为主的 Web 应用，更接近传统多页 MPA 的体验：通过 Next.js 的静态导出，在构建时生成多个页面，依赖浏览器的标准导航与页面加载完成 HR 与面试官的浏览路径。站点本身不包含实时互动或长连接能力，重点是稳定、可预测的浏览体验和良好的首屏加载表现。

### Technical Architecture Considerations

- 保持以静态导出（SSG）为主的架构，不依赖复杂的客户端状态管理或长连接。
- 允许正常的整页跳转（MPA 体验），在此基础上适度使用客户端增强交互，而不是强行构建完整 SPA。
- 页面结构与路由设计优先围绕「首页 → 项目列表 → 项目详情」和「首页 → 文章列表 → 文章详情」这两条主路径展开。

### Browser & Platform Support (browser_matrix)

- 目标支持「现代浏览器」：最新版 Chrome、Safari、Firefox、Edge。
- 移动端主要关注 iOS Safari 与 Android Chrome。
- 不考虑 IE 及其他旧版浏览器的兼容性，相关问题不作为 1.0 范围内的阻塞项。

### Responsive Design

- 所有核心页面（首页、项目列表、项目详情、文章入口）在桌面和常见移动端尺寸下需具备良好的自适应布局。
- 优先针对「桌面 + 手机竖屏」两档做优化，平板等中间尺寸以合理可用为主，不追求像素级精细适配。

### Performance Targets

- 核心页面在现代浏览器和正常网络环境下，应具备良好的首屏体验，不出现明显卡顿或异常缓慢加载。
- 构建时尽量控制每个页面的资源体积：按需加载组件与资源，避免为简单内容引入过于沉重的依赖。
- 对于图片与媒体内容，采用合适的尺寸与压缩策略，避免单个页面因媒体资源导致加载过慢。

### SEO Strategy

- SEO 重要性为中等偏低：主要流量来自你主动在简历、社交平台（如 LinkedIn、脉脉）中分享链接。
- 仍需保证基础 SEO 质量：
  - 每个页面有合理的 `<title>` 与 `<meta description>`。
  - 代表项目和文章详情页可被搜索引擎索引（不主动屏蔽）。
- 不在 1.0 阶段投入大量复杂 SEO 优化（如结构化数据、系统性长尾关键词运营），后续可视需要迭代。

### Accessibility Level

- 以「基础无障碍保证」（a11y-1）为目标：
  - 使用语义化 HTML 结构（例如合理使用 heading、列表、按钮等元素）。
  - 为关键图片与图标提供合适的 `alt` 文本。
  - 确保主要交互可以通过键盘操作完成，不出现完全无法访问的区域。
- 暂不追求完整覆盖 WCAG 所有细则，后续有时间可以在此基础上逐步增强。
