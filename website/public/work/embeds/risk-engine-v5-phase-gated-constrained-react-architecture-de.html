EVOLUTION

从 V2 到 V5：四代架构演进

版本 架构模式 核心问题 核心能力
V2 Single Agent 结构集中，难扩展 EvidenceAgent + CBR
V3 Prompt Chain 推理质量不足 4-step Chain Analyzer
V4 Query Rewriter 检索召回率低 Multi-query RAG + Strategy Gate
V5 Controlled ReAct 场景扩展成本高，主流程 if-else 膨胀 Registry + DomainAgent + MacroTool + PolicyArbiter
V4 痛点
新场景接入 = 修改主流程 if-else。证据获取与结论分析耦合过深。平台规则散落在多个地方，行为不一致。
V5 方案
四层职责分离 — 流程控制(Orchestrator)、场景知识(DomainAgent)、证据获取(MacroTool)、平台规则(PolicyArbiter)。新增场景 = 注册 Agent + Tool，不改主循环。
WHY IT MATTERS
V5 适合当前项目的原因，不在于它更像 AI，而在于它更适合长期扩风控场景。足够能扩，足够可控，足够能落地。

V2→V5 架构演进详解

V4 的三个核心短板

① 场景扩展退化为 if-else — 每接入一个新封禁场景（如骗绑、恶意注册），都需要在主流程中添加分支判断。场景越多，主循环越臃肿，回归测试成本线性增长。

② 证据获取与结论分析耦合过深 — V4 的 fetchAllEvidence() 一次性拉取全部证据，然后交给 ChainAnalyzer 串行分析。无法按需取证，也无法根据中间结论动态调整取证策略。

③ 平台规则缺少独立收口层 — 「保守覆盖」「强制人工」「评分覆盖」等平台级规则散落在 analyzer、handler、config 多个位置，新增规则时容易遗漏或产生冲突。

V5 四层分离方案

RiskOrchestrator

循环控制 · 步骤预算 · Tool 去重 · fallback 降级 · 结果收口。不包含任何场景知识，只管「跑几步、怎么跑、什么时候停」。

DomainAgent

场景级 planning，每个场景一个 Agent。Agent 看到当前 EvidenceState，决定下一步调哪个 Tool。场景知识封装在 Agent 的 Skill prompt 中。

MacroTool

取证能力做成可复用资产。FaceVerifyRule、GetAppealMaterial、SearchKnowledge 等 Tool 被多个 Agent 共享，避免重复实现。

PolicyArbiter

保守覆盖 · 强约束 · 评分覆盖 · 证据充分性覆盖。所有平台级规则在此统一收口，Agent 的 proposal 必须经过 Arbiter 审批。

V4 vs V5 代码对比

// V4 Anti-pattern: 场景知识耦合在主流程 func analyze(order Order) { if order.Scenario == "theft" { evidence := fetchAllEvidence(order) // 一次拉全部 return chainAnalyzer.Run(evidence) // 分析器承担流程控制 } else if order.Scenario == "fraud" { ... } } // V5 Pattern: 场景知识下沉到 Agent agent := registry.Resolve(input.Capability) // Registry 路由 action := agent.Plan(ctx, state) // Agent 自主取证 tool.Run(ctx, state) // 按需取证 arbiter.Arbitrate(ctx, state, proposal) // 统一规则收口

V5 核心架构详解

EvidenceState: 显式共享状态

type EvidenceState struct { OrderID string BlockUin uint64 // 跨工单历史查询 ToolRuns []ToolRun // 去重日志 ToolResults map[string]*Result // 缓存 CaseType CaseType // LLM 推理得出 SpamType string // 定性结论 CompletedPhases []string // 阶段追踪 Sufficiency string // LLM 自评 SufficiencyOverridden bool // 系统覆盖标记 ThoughtHistory []StepThought // 推理链 WorkingNotes []string // SkillRouter 信号 AppealHistory *AppealHistory // 跨工单历史 }

EvidenceState 是 Orchestrator 循环中唯一的可变载体。每个 Tool 只读写 State 中属于自己的字段，所有变更对 Langfuse trace 完全可见。

Registry: Agent 路由策略

① PreferredAgentID — 精确匹配：调用方显式指定 Agent ID，Registry 直接返回对应实例。用于测试和特殊场景强制路由。

② Capability — 遍历 Registry，按已注册 Agent 的 capability 标签匹配。多个匹配时取优先级最高者。

③ Default — 无匹配时降级到 LLMDrivenAgent，使用通用 prompt 和全量 Tool 集合。

④ aiClient 降级 — 若 aiClient 不支持 ToolCallingClient 接口，降级到 PrimaryRiskAgent（固定序列，无 ReAct 循环）。

架构不变量

不变量 1

EvidenceState 是唯一可变状态载体，无隐藏回调、无 goroutine 私有状态。

不变量 2

Registry 控制 Agent 解析，Agent 不自注册。所有 Agent 在启动时由 cmd/api 注入 Registry。

不变量 3

同一步内同名 Tool 不会被重复调用 — HasToolCall 去重机制防止 LLM 幻觉导致的重复取证。

不变量 4

阶段完成状态记录在 state.CompletedPhases，不在 Agent 记忆中。Agent 重启不丢失进度。

核心设计：所有状态显式流经 EvidenceState，支持完整 Langfuse 追踪和回放。

ORCHESTRATOR

Phase-Gated Constrained ReAct

核心改造：将纯 ReAct 循环拆为 3 阶段确定性流水线。取证阶段保持 LLM 自主，定性和裁决由 Orchestrator 强制驱动。

PROBLEM
问题：纯 ReAct 架构下，LLM 可随时调用 finalize 跳过 classification/verdict。线上 trace 显示 spam_type='其他'、决策质量下降。本质上是"让 LLM 决定是否执行定性和裁决"的架构问题。
Phase 1 · Evidence ReAct Loop — LLM 自主选择工具。退出条件：Sufficiency='sufficient' 或 tool budget 耗尽(6 calls)。系统级 ComputeSufficiency() 兜底。
Phase 2 · Classification Constrained — ForcedSkill=classification, AllowedTools=[SearchAuditRules, GetHistoricalCasePack], BlockFinalize=true。LLM 调用 finalize 会被拦截为 PhaseComplete。
Phase 3 · Verdict Deterministic — ForcedSkill=verdict, 仅允许 finalize。输出 UNBLOCK / KEEP_BAN / MANUAL_REVIEW + confidence + reasoning。
Evidence → sufficiency ✓ / budget ✗ → Classification → spam_type extracted → Verdict → finalize → PolicyArbiter
cp, ok := agent.(ConstrainedPlanner) // type assertion if !ok { return o.runLegacyReActLoop(...) } // 向后兼容 o.runEvidencePhase(ctx, cp, state) // Phase 1: 自由取证 o.runClassificationPhase(ctx, cp, state) // Phase 2: 强制定性 o.runVerdictPhase(ctx, cp, state) // Phase 3: 强制裁决
WHY IT MATTERS
不信任 LLM 的流程判断，只信任 LLM 的内容判断 — 证据阶段让 LLM 自主决定取什么证据，但阶段流转由 Orchestrator 确定性驱动。

Orchestrator · Phase-Gated Constrained ReAct 完整设计

1. ConstrainedPlanner 接口

type ConstrainedPlanner interface { PlanConstrained(ctx context.Context, state *model.EvidenceState, opts PlanOptions) (*model.AgentAction, error) } type PlanOptions struct { ForcedSkill string // 强制注入的 Skill（如 "classification"） AllowedTools []string // 工具白名单（空=全部允许） BlockFinalize bool // true=拦截 finalize 调用, 返回 PhaseComplete } // 向后兼容：不实现 ConstrainedPlanner 的 Agent 自动 fallback 到纯 ReAct

2. 三阶段执行引擎

Evidence Phase（自由 ReAct） LLM 自主选择 SkillRouter 推荐的 Skill，自由调用取证工具。退出条件：(1) state.Sufficiency='sufficient'（LLM + ComputeSufficiency 双重校验）；(2) tool budget 耗尽（默认 6）。系统级 ComputeSufficiency() 要求至少有 AppealMaterial + 1个内容/行为工具结果。LLM 自评 sufficient 但系统判定 insufficient 时，以系统为准。
Classification Phase（约束） ForcedSkill=classification, AllowedTools=[SearchAuditRules, GetHistoricalCasePack], BlockFinalize=true。LLM 必须输出【违规类型】（不含"正常"选项）。finalize 调用被拦截为 PhaseComplete，提取 state 后退出。最多 2 轮内循环。
Verdict Phase（确定性） ForcedSkill=verdict, 仅允许 finalize。LLM 输出 result_type + confidence + result_reason。UNBLOCK→SpamType 清空；KEEP_BAN→保留 SpamType；MANUAL_REVIEW→保留供人工参考。

3. 阶段转换条件

Evidence → Classification 转换条件: state.Sufficiency == "sufficient" AND ComputeSufficiency(state) == true OR tool_budget 耗尽 (forced transition) Classification → Verdict 转换条件: state.SpamType != "" (已提取违规类型) OR max_inner_steps 耗尽 (2 steps) Verdict → PolicyArbiter: LLM 输出 finalize action → DecisionProposal → Arbiter.Arbitrate()

4. Langfuse 可观测性

orchestrator.Run ├── phase.evidence {exit_reason, inner_steps, sufficiency} │ ├── agent.Plan[step=0] {skill_name, action_type} │ ├── macrotool.GetAppealMaterialPack │ └── agent.Plan[step=1] {...} ├── phase.classification {forced_skill, exit_reason, spam_type} │ └── agent.Plan[classification.step=0] ├── phase.verdict {forced_skill, exit_reason, result_type} │ └── agent.Plan[verdict.try=0] └── policy-arbiter.Arbitrate {input_result_type, override}

5. 核心设计思路

不信任 LLM 的流程判断，只信任 LLM 的内容判断。Orchestrator 控制"何时做什么"，Agent 决定"具体做什么"。

SkillRouter · 动态 Skill 路由与领域知识注入

1. Skill 路由算法

SkillRouter.SelectSkill(state) 算法: 1. 遍历所有已注册 Skill（embed.FS 自动发现） 2. 对每个 Skill 检查 trigger 条件: • tool_results_empty: true → 无任何工具结果 • has_tool_result + source → 指定工具有结果 • min_tool_count: N → 至少 N 个工具被调用 • sufficiency_check: sufficient → LLM 自评充分 • requires: [spam_type_set] → SpamType 已设置 • excludes: [phase:X] → 跳过已完成阶段 • case_types: [...] → CaseType 匹配（规划中） 3. 收集所有匹配的 Skill 4. 按 priority 排序（数字越大越优先） 5. 返回最高优先级的 Skill 6. 注入 Skill prompt 到 system message

2. 真实 Skill 内容示例

--- name: evidence-collection priority: 100 trigger: tool_results_empty: true --- ## 你的角色 你是企微反欺诈风控审核专家。当前工单尚无任何证据。 ## 取证策略 根据申诉内容判断取证优先级: - 明确恶意（色情/诈骗/赌博）→ 先 GetBehaviorRiskPack - 正常/不确定 → 先 GetAppealMaterialPack → GetCorpProfilePack ## <thought> 模板 <thought> 【当前证据评估】尚无证据 【证据充分度】insufficient 【下一步计划】调用 GetAppealMaterialPack 获取基础材料 </thought> ## 下一步建议 完成首轮取证后: - 如需审核规则参考 → SearchAuditRules - 如有历史案例需求 → GetHistoricalCasePack

3. Phase-Aware Thought 模板

改进方案
  • System prompt 的 <thought> 模板精简为通用 fallback
  • 各 Skill 在正文中定义阶段专属 thought 模板
  • LLM recency bias 自然覆盖 system prompt（无需代码改动）
  • content-review / behavior-review 新增【初步定性】字段，打破 CaseType 循环依赖
Skill = 领域知识的声明式资产。新增场景 = 新增 .md 文件，SkillRouter 通过 embed.FS 自动发现，无需改 Go 代码。

MacroTool · 统一证据获取接口

1. Tool 接口定义

type Tool interface { Name() string Description() string Parameters() map[string]any // JSON Schema for LLM Run(ctx context.Context, input Input) (*MacroToolResult, error) } type Input struct { OrderID string State *model.EvidenceState Arguments map[string]any // LLM 传参 } type MacroToolResult struct { ToolName string Summary string // LLM 可读摘要（~200 字） Facts []string // 结构化事实列表 MissingSources []string // 缺失数据源 RawData any // 原始数据（审计用） Confidence float64 // 0.0~1.0 }

2. RPC 适配层 (adapters.go)

// 每个 Evidence Tool 是一个 RPC 适配器: type GetAppealMaterialPackTool struct { fetcher func(ctx, orderID) (*MaterialResp, error) } func (t *GetAppealMaterialPackTool) Run(ctx, input) (*MacroToolResult, error) { resp, err := t.fetcher(ctx, input.OrderID) if err != nil { return nil, fmt.Errorf("RPC failed: %w", err) } return &MacroToolResult{ Summary: formatForLLM(resp), // 压缩到 ~200 字 Facts: extractFacts(resp), Confidence: 0.5, // RPC 数据是证据，不是结论 }, nil }

3. Tool 注册与 Setter 注入

依赖注入模式
  • 核心 tools 在 NewAuditHandler() 构造时注册
  • 可选 tools (CBR, Knowledge) 通过 Setter 运行时注入
  • SetCBRService() / SetKnowledgeClient() / SetQueryRewriter()
  • syncEvidenceAgentMacrotools() 同步到 Agent 的 tool 列表
  • 优点：不修改构造函数签名，支持灰度和动态绑定
新增取证能力 = 实现 Tool 接口 + 在 handler 注册。底层是 RPC、API 还是本地计算，LLM 完全无感。
POLICY ARBITER

PolicyArbiter 平台规则收口

Agent 提出 DecisionProposal，PolicyArbiter 用 5 条确定性规则做最终仲裁。AI 不可绕过，保证平台行为一致性。

Rule 1 好人分 Override — score≥0.85 且 Agent 判 KEEP_BAN → 覆盖为 UNBLOCK；score≤0.25 且 Agent 判 UNBLOCK → 覆盖为 KEEP_BAN。Logistic Regression 模型，权重按部署环境调优。
Rule 2 关键证据缺失 Force Reject — ≥2 项关键证据源缺失（申诉材料 / 企业画像 / 行为风控）→ 强制 MANUAL_REVIEW。宁可打回人工，不在信息不足时自动决策。
Rule 3 高风险+非管理员 Reject — spam_type∈{盗号,骗绑} 且不是管理员 → KEEP_BAN。高危场景对非管理员保持保守。
Rule 4 管理员无恶意 Unban — isAdmin=true 且无投诉/无拦截 → UNBLOCK。管理员在清白时优先解封。
Rule 5 Sanity Check — result_type 无效 → fallback REJECT。兜底规则防止 LLM 输出非法值。
// 5 条规则按顺序执行，前序规则覆盖后序 func (a *PolicyArbiter) Arbitrate(ctx, state, proposal) { proposal = a.applyGoodPersonScore(state, proposal) proposal = a.applyCriticalMissing(state, proposal) proposal = a.applyHighRiskNonAdmin(state, proposal) proposal = a.applyAdminCleanOverride(state, proposal) proposal = a.applySanityCheck(proposal) return proposal }
WHY IT MATTERS
LLM 做内容判断，PolicyArbiter 做平台规则。两层分离保证：即使 LLM 产生幻觉，确定性规则始终兜底。

KNOWLEDGE RETRIEVAL

BM25 审核规则知识库

从"全量注入"到"按需检索" — Agent 调用 SearchAuditRules 工具，BM25 排序返回 top-3 最相关规则。零外部依赖，纯 Go 实现。

PROBLEM

SOLUTION

Phase-Gated Constrained ReAct

BM25 DEEP DIVE
BM25 审核规则知识库 — 技术详解

BM25 算法原理

TF (Term Frequency)
查询词在文档中出现的频率。出现越多，相关性越高。BM25 对 TF 做饱和处理，避免高频词过度主导。
IDF (Inverse Document Frequency)
越稀有的词权重越高。"登录"出现在 80% 文档中权重低，"骗绑"只出现在 5% 文档中权重高。
BM25 = 改进的 TF-IDF
引入文档长度归一化（参数 b=0.75）和 TF 饱和度（参数 k1=1.2），短文档天然获得微弱加分。
中文 bigram 分词
将"多设备登录"拆为 ["多设", "设备", "备登", "登录"]。无需外部分词器，纯 Go 实现。
Go 纯实现
无 cgo 依赖，无外部搜索引擎。编译为单一二进制，部署零配置。

TRACE VALIDATION

Trace Validation Framework

声明式 YAML spec + Go CLI 验证器 — 消除目视 Langfuse trace 的主观性，让机器判 PASS/FAIL。 Agent-First 设计：spec 由 Claude Code 自动生成。

PROBLEM
Claude 跑线上测试后目视 Langfuse trace 判断"通过/不通过"，主观且不可重复。一次验证中声称 P0/P1/P2 全修复 ✅，但 spam_type='其他' 完全没发现。根因：验证标准在 Claude 的理解里，不在代码里。
SOLUTION
YAML spec 定义验证断言（12 种操作符），Go CLI 从 Langfuse API 获取 trace，逐条评估，输出结构化 PASS/FAIL 报告。severity 分级（critical 必须全过，warning 允许容忍度）。
YAML Spec → Go CLI → Langfuse API HTTP Basic Auth → Trace + Observations → Evaluator 12 ops → Verdict pass rate → Report Table / JSON → Exit Code 0/1/2
# testdata/specs/phase-gated-v5.yaml name: phase-gated-v5-validation assertions: - id: spam-type-not-other path: output.spam_type op: neq value: "其他" severity: critical - id: classification-phase-exists span: "phase.classification" path: metadata.phase_name op: eq value: "classification" severity: critical - id: verdict-exit-reason span: "phase.verdict" path: metadata.exit_reason op: eq value: "finalize" severity: critical thresholds: critical_pass_rate: 1.0 warning_pass_rate: 0.8
12 操作符： eq neq in not_in gt gte lt lte contains not_contains regex exists

WHY IT MATTERS
验证标准从"Claude 的理解"迁移到"代码/配置" — 声明式、可重复、机器判定。Exit code 语义化 (0=PASS, 1=FAIL, 2=ERROR) 支持 CI/CD 集成。

Trace Validation Framework — 详细架构

5 层验证架构

① Spec Layer — spec.go

YAML 解析 + 验证。TestSpec 包含 name / version / description / trace_id / assertions / thresholds。每个 Assertion 有 id / span / path / op / value / severity / message。12 种操作符在 validOperators map 中注册。

type TestSpec struct { Name string `yaml:"name"` Version string `yaml:"version"` Description string `yaml:"description"` TraceID string `yaml:"trace_id"` Assertions []Assertion `yaml:"assertions"` Thresholds Thresholds `yaml:"thresholds"` } var validOperators = map[string]bool{ "eq": true, "neq": true, "in": true, "not_in": true, "gt": true, "gte": true, "lt": true, "lte": true, "contains": true, "not_contains": true, "regex": true, "exists": true, }

② Fetch Layer — fetcher.go

Langfuse HTTP API 客户端。HTTP Basic Auth (public_key:secret_key)。GET /api/public/traces/{id} + GET /api/public/observations?traceId={id}。TraceData 包含 output/metadata + []Observation。toStringMap() 处理双重编码 JSON。

// FetchTrace: GET /api/public/traces/{traceID} // FetchObservations: GET /api/public/observations?traceId={traceID} func (f *Fetcher) FetchTrace(traceID string) (*TraceData, error) func (f *Fetcher) FetchObservations(traceID string) ([]Observation, error) // Auth: req.SetBasicAuth(f.publicKey, f.secretKey)

③ Evaluate Layer — evaluator.go

dot-notation 路径提取 (output.result.status → data["output"]["result"]["status"])。span 级过滤 (findObservation by name)。mergeOutputMetadata 构建统一查询结构 {output: {...}, metadata: {...}}。12 种操作符实现：数值比较归一化到 float64，字符串比较用 fmt.Sprintf。

func extractPath(data map[string]any, path string) (any, bool) { parts := strings.Split(path, ".") // dot-notation for _, part := range parts { m, ok := current.(map[string]any) current = m[part] } } func findObservation(obs []Observation, name string) *Observation func mergeOutputMetadata(obs *Observation) map[string]any

④ Report Layer — reporter.go

ComputeVerdict 按 severity 分别计算 pass rate。CriticalPassRate < threshold → FAIL。table 格式用 box-drawing 字符（┌├└），JSON 格式用 MarshalIndent。truncate() 处理 Unicode ([]rune)。

func ComputeVerdict(results []Result, thresholds Thresholds) Verdict { critPass := countBySeverity(results, "critical", true) critTotal := countBySeverity(results, "critical", false) critRate := float64(critPass) / float64(critTotal) // CriticalPassRate < threshold → FAIL if critRate < thresholds.CriticalPassRate { return FAIL } }

⑤ CLI Layer — main.go

5 步流水线: LoadSpec → LoadLangfuseConfig → FetchTrace + FetchObservations → EvaluateAssertions → ComputeVerdict → Report。Exit code: 0=PASS, 1=FAIL, 2=ERROR。

spec := LoadSpec(specPath) // Step 1 cfg := LoadLangfuseConfig(flags) // Step 2 trace, obs := fetcher.Fetch(traceID) // Step 3 results := EvaluateAssertions(spec, trace, obs) // Step 4 verdict := ComputeVerdict(results, spec.Thresholds) // Step 5 Report(verdict, format) // Output os.Exit(verdict.ExitCode) // 0=PASS, 1=FAIL, 2=ERROR

12 种断言操作符

操作符 类型 示例
eq 等值 value: "succeeded"
neq 不等 value: "其他"
in 枚举 value: [UNBLOCK, KEEP_BAN]
not_in 排除 value: [1, 2]
gt / gte 数值比较 value: 1 (tool_calls ≥ 1)
lt / lte 数值比较 value: 2 (inner_steps ≤ 2)
contains 包含 value: "classification"
not_contains 不包含 value: "error"
regex 正则 value: "^[0-9]+$"
exists 存在 (无 value，path 存在即 pass)

三层断言目标

Trace 级 — 无 span 字段

验证 trace 最终输出。断言直接作用于 trace 的 output/metadata。

- id: spam-type-exists path: output.spam_type # 直接访问 trace output op: exists severity: critical

Span 级 — 指定 span 名称

验证特定 span 的 metadata/output。通过 findObservation(name) 定位。

- id: classification-phase-exists span: "phase.classification" # 匹配 observation name path: metadata.phase_name op: eq value: "classification" severity: critical

跨 Span — Agent.Plan 级别

验证 Agent 在特定 phase/step 的决策行为。span 名包含动态参数。

- id: forced-classification-skill span: "agent.Plan[classification.step=0]" # 参数化 span path: metadata.selected_skill op: eq value: "classification" severity: critical
