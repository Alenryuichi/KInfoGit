# 基础原理
## 一、Transformer 中的 Q、K、V
先从大模型内部说起。在 Self-Attention 机制中：

+ **Query (Q)**：「我在找什么？」—— 当前 token 想要关注的信息
+ **Key (K)**：「我身上有什么标签？」—— 每个 token 用来被搜索的特征
+ **Value (V)**：「我实际携带的内容是什么？」—— 匹配成功后真正传递的信息

计算过程是：`Attention(Q, K, V) = softmax(QK^T / sqrt(d)) · V`

本质上就是一个**软性检索**：Q 和 K 做点积算相关度，再用这个相关度对 V 做加权求和。这和 Embedding 检索的思想是一脉相承的。

## 二、Embedding 知识库的分片（Chunking）
### 分片内容如何设置
分片策略直接决定了 Embedding 的质量和检索效果：

| 策略 | 特点 | 适用场景 |
| --- | --- | --- |
| 固定长度切分 | 简单粗暴，可能切断语义 | 快速原型 |
| 按段落/章节切分 | 保持语义完整性 | 结构化文档 |
| 语义切分 | 用模型判断语义边界 | 高质量需求 |
| 滑动窗口 + 重叠 | 避免边界信息丢失 | 通用场景 |
| 按问答对切分 | 每个 chunk 是一个完整 QA | FAQ 类场景 |

### 分片如何影响 Embedding 分布
这是最关键的点：

+ **Chunk 太大**：Embedding 变成多个主题的"模糊平均"，在向量空间中落在一个不明确的位置，谁都沾点边但谁都不精确匹配
+ **Chunk 太小**：丢失上下文，"机器学习"三个字的 embedding 无法告诉你它是在讲算法还是在讲职业
+ **Chunk 语义纯度高**：一个 chunk 只讲一件事，它的 embedding 在向量空间中的位置就很"尖锐"、很明确，容易被精准召回
+ **Chunk 语义混杂**：一个 chunk 讲了三件事，它的 embedding 会被"拉扯"到几个语义簇的中间地带，导致哪个 query 都能勉强匹配，但都不是最佳结果

你可以把向量空间想象成一张地图：语义纯的 chunk 是精确的地标，语义杂的 chunk 是模糊的区域。Query 来了就是在地图上找最近的点。

## 三、Q2Q、Q2A、A2A —— 这是 Embedding 训练和检索的核心范式
这组概念描述的是：**在向量空间中，你在比较什么和什么？**

### Q2Q（Query-to-Query）
+ **含义**：用问题匹配问题
+ **知识库存的是**：问题（每个问题关联一个答案）
+ **检索逻辑**：用户问了一个新问题 → 找到最相似的已有问题 → 返回那个问题对应的答案
+ **典型场景**：FAQ 系统、客服机器人
+ **优势**：问题和问题在语义空间中天然接近（都是疑问句式、都在问同一件事），匹配率高
+ **例子**：用户问"怎么退款" → 匹配到"如何申请退款" → 返回退款流程

### Q2A（Query-to-Answer）/ Q2D（Query-to-Document）
+ **含义**：用问题匹配答案/文档
+ **知识库存的是**：答案或文档片段
+ **检索逻辑**：用户问了一个问题 → 直接在文档块中找语义最接近的
+ **典型场景**：通用 RAG、文档问答
+ **核心难点**：这是**非对称匹配** —— 问题和答案的表述形式完全不同。"Python 怎么读文件？" 和 "`open('file.txt', 'r')` 可以用来读取文件" 说的是一件事，但文本形式差异很大
+ **对模型要求**：Embedding 模型必须在训练时就见过大量 (question, answer) 对，学会把问题和对应答案映射到向量空间的相近位置

### A2A（Answer-to-Answer）/ D2D（Document-to-Document）
+ **含义**：用文档匹配文档
+ **检索逻辑**：给一段文本 → 找语义相似的其他文本
+ **典型场景**：去重、聚类、相似文章推荐、知识图谱构建
+ **特点**：对称匹配，两边是同类型文本

### 直观对比
```plain
Q2Q:  "怎么退款？"  →  匹配  →  "如何申请退款？"         (问题 vs 问题)
Q2A:  "怎么退款？"  →  匹配  →  "点击订单详情，选择退款"   (问题 vs 答案)
A2A:  "退款流程..."  →  匹配  →  "退货退款步骤..."        (文档 vs 文档)
```

## 四、Query 该如何设置？
这取决于你的知识库用的是哪种范式：

1. **如果知识库存的是文档（Q2A 模式）**：
    - Query 就是用户的自然语言问题
    - 但可以做 **Query Rewriting**：用 LLM 把用户模糊的问题改写成更明确的检索 query
    - 也可以做 **HyDE**（Hypothetical Document Embedding）：让 LLM 先生成一个"假想答案"，用这个假想答案去检索，本质上把 Q2A 问题转化成了 A2A 问题
2. **如果知识库存的是问题（Q2Q 模式）**：
    - 建库时需要为每个知识点预设多种问法
    - Query 就是用户原始问题，直接匹配
3. **混合策略**：
    - 对每个文档 chunk，用 LLM 生成若干"可能会被问到的问题"
    - 知识库同时存原始 chunk 和生成的问题，检索时两路召回再融合

## 五、这些概念之间的深层联系
Transformer 内部的 Q·K 点积注意力，和 Embedding 检索中的 query·document 余弦相似度，**本质上是同一个操作**——在高维空间中做相似性匹配。区别在于：

+ Attention 是**模型内部的软检索**，每一层都在做
+ Embedding 检索是**模型外部的硬检索**，在知识库上做一次

理解了这个统一视角，很多设计决策就变得清晰了：分片质量决定了"被检索方"的信号纯度，query 设计决定了"检索方"的指向精度，而 Q2Q/Q2A/A2A 的选择决定了这两方是否在同一个"语义子空间"中对话。
