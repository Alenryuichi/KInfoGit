## 一、引言：为什么是 Harness？
2026 年的软件工程正在经历一场不可逆的范式转移。这场转移的核心不在于模型变得更聪明——GPT-5.1、Claude Opus 4.6、Gemini 2.5 的智力已经是公共品——而在于**谁能为 AI Agent 构建最严密的"物理法则"，让它在百万行代码库中不发疯、不遗忘、不破坏**。

这套"物理法则"有一个精确的工程术语：**Harness Engineering**（脚手架工程 / 工程化治理）。

**"Harness"** 这个词的原意是"马具/挽具"——不是马本身，也不是骑手的技术，而是**套在马身上、让马的力量可控且有方向的物理装置**。这个隐喻精准地描述了我们与大模型之间的关系：模型是那匹力量无穷的马，而工程师的工作，是设计让这匹马跑得快、跑得准、不翻车的挽具系统。

本文将从概念溯源、理论框架、工业级实证、五层解剖、安全治理、多 Agent 协作等维度，全面解析 Harness Engineering 在 2026 年的最新范式。

## 二、概念溯源：从 Prompt Engineering 到 Harness Engineering 的三阶段演进
AI 工程的发展并非简单的替代，而是层层递进的嵌套关系。理解 Harness Engineering 的定位，必须先理解它在范式演进光谱上的坐标。

### 2.1 第一阶段：Prompt Engineering（提示词工程）
**核心关注："怎么说"。** 通过角色扮演、思维链（Chain of Thought）、Few-Shot 等技巧优化单次对话质量。这是 2023-2024 年的主流范式。

**局限性：** 存在根本性天花板——不稳定、不可扩展、缺乏记忆和系统观。换个场景，精心调优的 Prompt 可能就失效了。正如 Gartner 在 2025 年 10 月的报告中所述："提示工程正逐渐淡出并被工具和模板所取代，企业面临更深层的问题：如何赋予 AI 正确的情境感知能力。"[^1]

### 2.2 第二阶段：Context Engineering（上下文工程）
**核心关注："给什么信息"。** 通过 RAG（检索增强生成）、历史对话管理、工具调用（Tool Use）、MCP（Model Context Protocol）等，动态组装信息，让 AI 获取当前任务所需的知识。

Gartner 将其定义为：

> **设计和构建相关的数据、工作流和环境，使 AI 系统能够理解意图、做出更好的决策，并交付符合企业情境的预期结果——而无需依赖手动提示。**[^1]
>

**局限性：** 虽然解决了"知道什么"，但无法阻止 AI "做不该做的事"。Agent 缺乏约束，容易在执行中走偏、过度操作。上下文工程解决了信息供给，但没有解决行为控制。

### 2.3 第三阶段：Harness Engineering（脚手架工程）
**核心关注："在什么环境下做事"。** 不再局限于语言或知识，而是构建一个包含**约束、反馈、可观测性和反熵机制**的完整运行时系统。这是 2026 年最高的演进层次。

这个术语在 2025 年底由 HashiCorp 创始人 **Mitchell Hashimoto** 率先提出（"Engineer the harness, not the model"），随后被 **OpenAI Codex 团队**在 2026 年 2 月 11 日发布的技术博客 _"Harness engineering: leveraging Codex in an agent-first world"_ 中正式系统化[<sup>2]。Thoughtworks 的 Distinguished Engineer </sup><sup>**Birgitta Böckeler**</sup><sup> 随即在 Martin Fowler 网站上发表了深度分析，将其归纳为三大支柱：</sup><sup>**上下文工程、架构约束和反熵垃圾回收**</sup><sup>[</sup>3]。

| 维度 | Prompt Engineering | Context Engineering | Harness Engineering |
| :--- | :--- | :--- | :--- |
| **核心关注** | 怎么说 | 给什么信息 | 在什么环境下做事 |
| **作用对象** | 单次对话 | 信息供给链路 | 整个运行时系统 |
| **控制方式** | 概率引导 | 知识注入 | 确定性约束 + 反馈闭环 |
| **可扩展性** | 低（场景敏感） | 中（需持续维护管道） | 高（规则编码一次、持续执行） |
| **典型产出** | 一段好回答 | 一个知识增强的回答 | 一个可靠的工程系统 |

三者的关系是嵌套而非替代：**Harness Engineering 包含 Context Engineering，Context Engineering 包含 Prompt Engineering**。真正的高手不是在某一层做到极致，而是在三层之间游刃有余。

## 三、理论框架：Birgitta Böckeler 的三大支柱
Thoughtworks Distinguished Engineer Birgitta Böckeler 在 2026 年 2 月 17 日于 Martin Fowler 网站上发表的深度分析[^3]，将 OpenAI 的实践归纳为三个混合了**确定性方法**（传统编程）和 **LLM 方法**（大模型）的支柱。这是目前业界对 Harness Engineering 最权威、最系统的理论框架。

### 支柱一：上下文工程（Context Engineering）
这不是简单的"写个好 Prompt"，而是构建一个**持续增强的知识基础设施**。核心洞察是：**代码设计本身就是最重要的上下文。**

上下文工程的目标是确保 AI Agent 在决策时拥有最全面、最新的情境信息。它包含两个维度：

**静态上下文**——项目的不变法则（技术栈约束、架构分层规则、命名规范等），锁定在 Context Window 顶部以命中 Prompt Cache。

**动态上下文**——当前任务的进度、运行态信号（日志、指标、浏览器截屏、DOM 快照等），按需注入。

把所有东西塞进一个文件（2025 年的常见做法）会导致严重的"Token 肥大症"和缓存失效。

### 支柱二：架构约束（Architectural Constraints）
这是 Harness 最"硬核"的部分。通过双重监控机制来强制执行架构规则：

**LLM 智能体监控**——利用 AI 检查代码是否符合架构意图。

**确定性自定义 Linter**——编写传统的、规则明确的代码检查工具（不依赖概率模型）。

**结构性测试**——使用如 ArchUnit 等框架，测试代码的架构结构（模块依赖、分层规则）。

Böckeler 指出，为了获得可信赖的、大规模的 AI 生成代码，必须**牺牲"生成任何东西"的灵活性**，将解决方案空间收缩到特定的架构模式内。未来可能会出现专门为了易于被 AI 维护而设计的代码库结构——"AI 友好的拓扑结构"。

### 支柱三：反熵垃圾回收（Anti-Entropy Garbage Collection）
AI Agent 有一个致命弱点：**它会复制代码库中已有的模式，包括坏模式**。一个不优雅的写法如果在代码库中出现过，Agent 就会不断复制它，导致代码腐化加速。

Böckeler 将第三大支柱定义为"对抗熵增"——通过自动化机制持续修复文档不一致和架构违规，防止代码库随时间推移而腐化。这就像编程语言中的垃圾回收（GC）：技术债务像高息贷款，小额持续偿还比痛苦地一次性解决要好。

## 四、工业级实证：三个标杆案例
### 4.1 OpenAI Codex 团队：100 万行零手写代码
OpenAI Codex 团队用自身实践给出了最有力的证明[^2]。从 2025 年 8 月下旬开始，初期由 **3 名工程师**推动（后扩大至 7 名），在 5 个月内利用 Codex 从空白 Git 仓库产出了**超过 100 万行代码**，构建了一整套软件产品的内部 Beta 版——涵盖应用逻辑、基础设施、工具链、文档、测试、CI 配置、可观察性和内部开发者工具。

**没有一行代码是人工手写的。** 甚至连指导 Agent 如何工作的 `AGENTS.md` 也是 Codex 自己编写的。

关键数据如下：

| 指标 | 数值 |
| :--- | :--- |
| **代码规模** | 约 100 万行 |
| **效率提升** | 约 10 倍（相较手工编写） |
| **团队规模** | 初期 3 人 → 后期 7 人 |
| **PR 总量** | 约 1,500 个 |
| **日均吞吐** | 每位工程师每天 3.5 个 PR |
| **Agent 单次工作时长** | 经常超过 6 小时（通常在人类睡眠时间） |

**他们不是靠更聪明的模型做到的，而是靠了一套极其严密的 Harness 系统。** 当 Agent 遇到困难时，工程师不直接写代码修复，而是把这视为一个**信号**——识别出 Harness 中缺失的工具、约束或文档，然后让 Codex 本身去编写修复方案并反馈到代码库中。这形成了一个自增强的闭环。

OpenAI 团队指出，目前最困难的挑战不再是写代码，而是**"设计环境、反馈循环和控制系统"**。

### 4.2 Anthropic：16 个 Claude 并行构建 C 编译器
2026 年 2 月 5 日，Anthropic 安全团队研究员 Nicholas Carlini 发表了一篇里程碑式的工程博文[^4]：他将 16 个 Claude Opus 4.6 实例投入并行计算，让它们在**没有人类实时监督**的情况下自主组队协作，从零用 Rust 编写了一个能编译 Linux 6.9 内核的 C 编译器。

关键数据如下：

| 指标 | 数值 |
| :--- | :--- |
| **代码规模** | 约 10 万行 Rust 代码 |
| **API 成本** | 约 20,000 美元 |
| **Agent 会话数** | 近 2,000 个 |
| **Token 消耗** | 20 亿输入 + 1.4 亿输出 |
| **时间跨度** | 约 2 周 |
| **测试通过率** | GCC torture test suite 达 99% |

这个案例的 Harness 工程亮点在于：

**容器化隔离**——每个 Agent 运行在独立的 Docker 容器中，拥有独立的 `/workspace` 工作目录。所有 Agent 共享一个裸 Git 仓库 `/upstream` 作为代码同步中心。这种架构防止了 Agent 之间的工作环境互相干扰，也防止了 Agent 意外破坏宿主机。

**基于文件锁的简易协调**——Agent 通过在 `current_tasks/` 目录下写入文本文件来声明任务锁定。如果两个 Agent 尝试锁定同一文件，Git 的同步机制会强制第二个 Agent 选择不同任务。没有中央"管理者"Agent，每个 Agent 完全自主决策。

**角色专业化**——除了通用开发 Agent，还设置了代码整合 Agent（合并重复代码）、性能优化 Agent、代码审查 Agent、文档维护 Agent 等专业化角色。

**针对 LLM 特性的环境设计**——测试工具仅输出少量关键信息，详细日志存入文件；错误信息包含 "ERROR" 关键字以便 grep 查找；提供 `--fast` 选项仅运行 1%-10% 的随机抽样测试。这些都是典型的 Harness 思维：为 Agent 的认知特性量身定制工程环境。

### 4.3 Stripe：工业级 AI PR 流水线
Stripe 建立了名为 **Minions** 的 AI PR 流水线体系，实现了每周合并超过 **1,300 个**由 AI 编写的 Pull Request。其 Harness 设计包含三个关键组件：

**预热 Devbox 隔离环境**——每个 AI 任务在独立的预置沙箱中运行，确保互不干扰、安全可控。

**Toolshed 中心化工具访问**——禁止 Agent 通过野路子直连外部系统，必须通过统一的工具层进行所有外部交互。

**混合编排模式**——并非将所有决策交给 AI，而是将需要严格可预测的步骤设为"确定性节点"（传统代码），需要灵活判断的交给"Agent 节点"（LLM），取长补短。

### 4.4 LangChain：环境优化的力量证明
LangChain 在 Terminal Bench 2.0 基准测试中提供了一个极具说服力的对照实验：**没有更换模型**，也没有修改 Prompt，仅仅优化了 Agent 的运行环境——完善文档、增强验证机制、改进追踪系统——排名从第 **30** 位直接跃升至第 **5** 位，得分从 52.8% 提升至 66.5%。

这个案例以最直接的方式证明了 Harness Engineering 的核心命题：**同样的"马"，换了一套更好的"马具"（环境），成绩发生了质的改变。**

## 五、Coding Agent 的五层解剖学
要让一个 Coding Agent 在庞大的代码库中不发疯、不遗忘、不破坏，必须为它组装以下五个核心器官。这不是一个"聊天窗口"，而是一个拥有完整解剖学结构的**"赛博生物"**。

### 5.1 第一层：宪法（The Constitution）—— `CLAUDE.md` / `AGENTS.md`
**定位：绝对静态的底线法则（Always-On Context）**

宪法是唯一一个在每个 Session 启动时被强制锁定在 Context Window 最顶部的文档，以确保 100% 命中 Prompt Cache。它只定义跨越整个项目生命周期的绝对真理。

**2026 最新范式：极简宪法与 JIT 注入**

OpenAI 的实践经验表明[^2]，一个巨大的 `AGENTS.md` 文件会面临四重困境：

1. **情境稀缺**——挤占了任务和代码的上下文窗口。
2. **指导无效**——当一切都很"重要"时，Agent 只会在本地进行模式匹配，无法导航。
3. **腐烂过快**——变成陈旧规则的坟场。
4. **难以核实**——无法机械检查其新鲜度和所有权。

正确做法是将 `AGENTS.md` 变成简短的**目录**（约 100 行），指向代码仓库中结构化的知识库。OpenAI 团队的原话是：**"给 Codex 一张地图，而不是一本 1,000 页的说明书。"**

其具体目录结构如下：

```plain
AGENTS.md          ← 入口索引（~100行）
ARCHITECTURE.md    ← 架构总览
docs/
├── design-docs/       # 设计文档（含验证状态、核心理念）
│   ├── index.md
│   ├── core-beliefs.md
│   └── ...
├── exec-plans/       # 执行计划（视为一等公民）
│   ├── active/
│   ├── completed/
│   └── tech-debt-tracker.md
├── generated/        # 生成内容（如数据库模式快照）
│   └── db-schema.md
├── product-specs/    # 产品规格
│   ├── index.md
│   └── ...
├── references/       # 外部引用文本化
│   ├── design-system-reference-llms.txt
│   └── ...
├── DESIGN.md
├── FRONTEND.md
├── PLANS.md
├── PRODUCT_SENSE.md
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md
```

**关键规则：**

**渐进式披露**——Agent 从一个小而稳定的入口开始，被引导去查看特定文档。不是一次性灌输，而是按需导航。

**严格验证**——使用 Linter 和 CI 作业验证知识库的更新、交叉链接和结构正确性。

**自动维护**——定期运行"doc-gardening"智能体，扫描并修复过时或废弃的文档。

**宪法即代码**——不要用模糊的自然语言写宪法。用 Markdown 的 Checklist 甚至 JSON Schema 定义，确保机器和人类都能精确解读。

### 5.2 第二层：潜意识与技能（The Subconscious）—— Skills
**定位：按需加载的程序性记忆（On-Demand Procedural Knowledge）**

如果把所有的操作规范都写在宪法里，AI 会"精神分裂"。Skills 是存放在特定目录下的 Markdown 文件（如 `.claude/skills/`），每个 Skill 详细规定了某个特定工作流的每一步。

**与宪法的本质区别：按需唤醒。** 宪法是 Agent 每秒钟都在思考的"重力法则"，而 Skill 是一本"字典"。当你在终端输入特定命令（如 `/tdd-cycle`），或者 Agent 判定当前任务需要该技能时，才会把相关 Token 动态加载到上下文中。

**实战收益：** Agent 在写 UI 的时候，脑子里完全不需要装着"如何压测 Go 接口"的技能上下文。这是极其重要的 Token 经济学优化。

### 5.3 第三层：长时记忆（The Hippocampus）—— Memory
**定位：跨越时空的防呆机制（Cross-Session Persistent Context）**

没有记忆的 Agent，就像一条只有 7 秒记忆的金鱼，每天都在踩昨天踩过的坑。Karpathy 在论述 Software 3.0 时明确指出：LLM 患有"顺行性遗忘症"，无法在长时间内保留上下文或从过去的交互中学习[^5]。

**2026 最新范式：从文件到 RAG 图谱**

放弃在根目录维护一个名为 `MEMORY.txt` 的无结构文件。全面接入 Memora 或 OpenMemory 这样的本地向量/图谱数据库。

**核心机制：**

**自动墓碑化（Automated Tombstoning）**——业务逻辑是会变的。当 Agent 废弃了一个旧的 API 接口，Memory 系统不会物理删除它，而是打上 `[STALE]` 标签。下次 Agent 搜索时，系统会告诉它："之前有这个接口，但上周被你干掉了，现在用新的。"

**语义直觉**——Agent 在面对莫名其妙的编译报错时，不需要精准输入关键词，直接调用记忆搜索接口，底层向量库就能凭借语义相似度，把两个月前解决同类报错的"心得"喂给它。

### 5.4 第四层：神经反射弧（The Nervous System）—— Hooks
**定位：绝对确定性的物理拦截（Deterministic Interception）**

大模型（LLM）是概率模型，它有可能会"发癫"。你不能在 Prompt 里哀求它："求求你，提交代码前一定要跑一下 Lint。"这是 2025 年的做法。**在 2026 年，不要用 Prompt 解决工程问题，用 Hooks。**

这个洞察与 arXiv 论文 _"Professional Software Developers Don't Vibe, They Control"_（2025 年 12 月，N=99 调查 + N=13 观察）[^6]的核心发现高度一致：**专业开发者不是在与 AI "vibing"，而是在积极控制它**。他们利用深厚的技术功底来指导和限制 Agent 的行为，用确定性约束而非概率性期望来保证质量。

**双重 Hook 体系：**

**Git Hooks（人类级拦截）**——传统的 `pre-commit`，用来兜底，防止 Agent 生成的垃圾代码污染主分支。

**Agent Hooks（Agent 级拦截——2026 杀手锏）**——这是直接植入 AI 思考循环里的挂钩，基于事件的 `PreToolUse` / `PostToolUse` 机制：

+ **PreToolUse 拦截**：假设 Agent 决定调用 `edit_file` 工具修改一个极其核心的鉴权中间件。钩子触发一个本地脚本，检测到修改敏感文件，直接返回拒绝信号，并强行注入指令："修改鉴权逻辑必须先生成一份 Architectural Decision Record (ADR)"。Agent 会被直接打断，乖乖去写文档。
+ **PostToolUse 拦截**：Agent 修改完代码文件后，钩子强制静默运行格式化工具和静态分析。如果失败，直接把报错返回给 Agent 强迫重修。这一切对人类是完全隐形的，你只看到最后完美的结果。

### 5.5 第五层：终极裁判所——The Harness Runtime
**定位：闭环自愈的物理沙盒（The Runtime Arena）**

这是 2026 年 Agentic Engineering 的最高层。Harness 不是一个工具，它是一个**"系统"**——包含了前四层的所有能力，加上运行时环境、验证反馈循环和反熵机制的完整闭合。

OpenAI 展示的高级自主流程[^2]清晰地说明了 Harness Runtime 的运作方式：

1. 验证代码库当前状态
2. 复现已报告的漏洞
3. 录制故障演示视频
4. 实施修复
5. 运行应用验证修复
6. 录制解决方案视频
7. 打开 PR
8. 回应反馈（Agent 和人类）
9. 检测并修复构建故障
10. 仅在需要判断时交由人工处理
11. 合并更改

整个流程中，人类只在第 10 步被"升级通知"，其余全部由 Harness 系统自动驱动。

## 六、架构约束的工业级实践
### 6.1 OpenAI 的分层架构与品味不变式
OpenAI 为其百万行代码库设计了一套强制的架构层级[^2]：

```plain
Types → Config → Repo → Service → Runtime → UI
```

**依赖方向是物理强制的**——任何违反该架构的代码都会被自动阻止。每个业务域（如 App Settings）内部遵循相同的分层结构，依赖只能单向流动，不可逆向。横切关注点（认证、连接器、遥测、功能标志等）必须通过单一的显式接口进入——**Providers**。任何通过其他方式进入的横切逻辑都被禁止。

更重要的是，他们把工程师的"品味"编码化了——这些被称为**"品味不变式"（Taste Invariants）**：

| 不变式类型 | 具体规则 |
| :--- | :--- |
| **数据解析** | 强制在边界处解析数据形状，而非内部随意使用 |
| **日志记录** | 强制结构化日志记录 |
| **命名约定** | 模式和类型的命名规范严格统一 |
| **文件限制** | 文件大小设置上限 |
| **依赖选择** | 倾向于完全可内化于仓库推理的依赖项 |

一个极具说明性的案例是：团队没有引入通用的 `p-limit` 包来做并发控制，而是实现了一个自定义的并发 map 辅助函数。原因在于它与 OpenTelemetry 集成，测试覆盖率 100%，行为完全符合预期，**避免了公共库中不透明行为对 Agent 的困扰**。

这些规则通过自定义 Linter 和结构测试机械地强制执行，**错误信息中直接包含修复指令**——不是告诉 Agent"你错了"，而是告诉它"你应该这样改"。这是一个极其精妙的设计：Lint 报错信息本身就是注入 Agent 上下文的补救指南。

### 6.2 Phodal 的 Fitness Function 体系
Phodal 在 2026 年 3 月的实践中进一步将架构约束具体化为 **Fitness Function**（适应度函数）体系[^7]——这不是传统的架构质量检查，而是**控制 Agent Loop 退出条件的核心机制**。

**核心问题：** 在 Agent Loop 中，Agent 只能理解局部信号（命令成功、无报错），但无法判断系统层面的"真正完成"。功能路径可以跑通，但负向路径没有覆盖；接口已经修改，但契约没有同步；测试数量增加了，但关键不变量并没有被验证。

**Fitness Function 的解法：**

**规则即代码**——Fitness 规则以 Markdown Frontmatter 编写，存放在 `docs/fitness/*.md`，兼顾人机可读。示例如下：

```yaml
---
dimension: testability
weight: 14
threshold:
  pass: 80
  warn: 70
metrics:
  - name: ts_test_pass
    command: npm run test:run 2>&1
    pattern: "Tests\\s+(\\d+)\\s+passed"
    hard_gate: true
  - name: rust_test_pass
    command: cargo test --workspace 2>&1
    pattern: "test result: ok"
    hard_gate: true
---
```

**统一执行器**——`fitness.py` 扫描规则文件，解析配置，运行命令，根据输出模式判断通过/失败。收回了规则的解释权，消除了"偶发问题"或"临时忽略"的模糊空间。

**证据文件**——记录验证状态（VERIFIED / TODO / BLOCKED），覆盖场景（CRUD 闭环、边界条件等），充当"工程账本"。

**硬门禁**——与评分体系不同，Hard Gate 失败会直接阻断流程。这是 AI 时代的 Definition of Done，明确区分"质量折损"和"流程终止"，防止 Agent 将"还不错"误解为"可以结束"。

| Gate 命令 | 阈值/条件 |
| :--- | :--- |
| `npm run test:run` | 100% |
| `cargo test --workspace` | 100% |
| `npm run api:check` | pass |
| `npm run lint` | 0 errors |

**防 API 语义漂移**——在多后端系统（如 Next.js + Rust/Axum）中，Agent 容易导致各实现间失去同构关系。解决方案是将 **OpenAPI 文件**作为单一事实来源，所有实现必须围绕同一组 endpoint 收敛，通过 `api_parity_check` 进行契约一致性校验。

## 七、反熵垃圾回收的工程实现
### 7.1 为什么需要反熵？
软件开发中自然倾向于混乱（熵增）。AI Agent 的参与非但没有缓解，反而**加速了这一过程**：Agent 会复现代码仓库中已存在的模式，包括那些不均衡或不理想的模式。一个"还凑合"的写法如果在代码库中出现过，Agent 就会不断复制它，形成"AI 生成残渣"。

OpenAI 团队的早期做法是每周五（占 20% 时间）手动清理这些残渣，但很快发现**这无法扩展**[^2]。

### 7.2 黄金原则的编码化
OpenAI 的解决方案是将带有主观意见的机械规则编码到仓库中，形成"黄金原则"：

+ 倾向于使用共享实用程序包而非手写辅助工具。
+ 禁止"YOLO 式"数据探测，必须验证边界或使用类型化 SDK。
+ 所有结构性重复必须收敛为抽象。

然后设置**后台 Codex 任务**定期自动运行：扫描偏差、更新质量等级、发起有针对性的重构 PR。大多数清理 PR 可以在一分钟内完成审查并自动合并。

Böckeler 将其总结为一个精妙的类比[^3]：

> **这就像编程语言中的垃圾回收（GC）。你不需要手动管理内存，但你需要设计一套自动化的回收机制来防止泄漏。在 AI 时代，"泄漏"的不是内存，而是架构意图和代码品味。**
>

### 7.3 文档维护的自动化
OpenAI 的知识库不是一次性写完就束之高阁的。他们使用 CI 和 Linter 检查文档新鲜度，定期运行"doc-gardening"Agent 自动扫描并修复过时或废弃的文档。这种机制确保了代码库的"记录系统"——所有隐性知识的显性化存储——始终保持准确和可导航。

## 八、上下文工程的深度实践
### 8.1 提升可读性：让应用对 Agent 直接可见
随着吞吐量增加，瓶颈从"写代码"变为"人类 QA"。OpenAI 的对策是让应用程序对 Codex **直接可读**[^2]。

**Git Worktree 支持**——应用程序配置为支持 `git worktree`，Codex 可以为每次更改启动并驱动一个独立的实例，避免不同任务之间的环境污染。

**浏览器自动化**——接入 Chrome DevTools 协议，Codex 可以处理 DOM 快照、截图和导航，从而复现错误、验证修复并直接推理 UI 行为。Agent 甚至可以**录制演示故障的视频**，修复后再录制解决方案视频。

**本地可观测性堆栈**——为每个 worktree 提供临时的本地可观测性堆栈。Codex 可以使用 LogQL 查询日志，使用 PromQL 查询指标。支持类似"确保服务启动在 800ms 内"或"关键用户旅程跨度不超过两秒"的提示验证。

这些做法的本质是：**将 Agent 从"只读代码"升级为"读运行态"。** Agent 不再是一个只能看到源码文本的盲人，而是一个能看到浏览器界面、能读懂系统日志、能感知性能指标的"全感知"开发者。

### 8.2 知识显性化原则
OpenAI 团队的一个核心原则是：**存在于 Google Docs、Slack 或个人脑子里的知识，对 AI 来说是不存在的。**

必须将所有知识——包括团队架构方向、决策逻辑、设计历史——显式化、结构化、版本化，并放入代码仓库。不仅是给 AI 看，也是为了新人入职的一致性。代码仓库不再仅仅是"放代码的地方"，而是整个产品团队的**唯一记录系统（Single Source of Truth）**。

## 九、合并理念的根本转变：纠错成本 vs 等待成本
传统软件工程的合并理念建立在一个隐含假设上：**代码产出慢，审查阻塞的时间成本可控**。在人工编码时代，一个 PR 可能需要几天才能准备好，等待审查的几个小时无伤大雅。

AI Agent 彻底颠覆了这个等式。当 Agent 的代码吞吐量达到每人每天 3.5 个 PR 时，传统的阻塞式门禁（merge gate）会将"等待"转化为主要成本。OpenAI 的核心洞察是[^2]：

> **在纠错成本低、等待成本高的系统里，快速迭代优于严格阻塞。**
>

具体做法包括：

| 传统模式 | Harness 模式 |
| :--- | :--- |
| 所有 PR 必须人工审查 | 绝大多数审核调整为"Agent 对 Agent" |
| 测试偶发失败阻塞合并 | 通过后续重跑解决，不阻塞进展 |
| PR 生命周期长（天级） | PR 生命周期极短（小时级） |
| 门禁越多越安全 | 减少阻塞合并门，用反熵机制兜底 |

这不是说质量不重要，而是**严谨性的重新定位**：从人工代码审查转移到环境设计、控制系统构建和自动化护栏编写上。用"快速试错、快速修正、持续清理漂移"取代传统的"先审后合"。

## 十、安全治理：Harness 的防御视角
### 10.1 AI 生成代码的安全挑战
Phodal 在 2026 年 3 月的安全分析中指出了一个关键洞察[^8]：**AI 并没有发明新的漏洞类别**。通过对多个开源项目运行 Codex Security 安全扫描后发现，最常见的安全问题依然是传统漏洞——未认证 API、命令注入、SSRF、HTML 渲染未清洗内容以及权限绕过。

真正的新风险在于两个方面：

**缺陷扩散机制的变化**——在传统模式中，错误可能是个体疏忽；但在 AI 辅助开发中，模型会将某种"看起来合理"但实际有缺陷的实现模式，**系统化地复制**到多个模块或仓库中。一个开发者的"忘了做输入验证"变成了模型级别的"从不做输入验证"。

**工具配置的安全风险**——扫描发现配置文件中存在 `--dangerously-skip-permissions`、`--allow-all-tools` 等参数，这些在本地开发时为了便利而设置的选项，在生产环境中可能导致权限控制被完全绕过。

### 10.2 Harness 的安全防御架构
**明确约束**——将系统的结构、规则和边界明确表达出来，使 AI 能够理解和遵守。不依赖 Agent 的"常识"来遵守安全规范。

**收敛危险能力**——将危险操作收敛到少数受控接口（如统一的外部请求客户端、安全的渲染器）。禁止 Agent 直接调用底层危险 API。

**自动化安全检查**——在代码进入主干前识别高风险模式。Codex Security 不仅做静态分析，还结合数据流追踪，从输入源（如 HTTP 请求）追踪到危险操作（如 Shell 执行），还原完整的攻击路径，并在沙箱中生成 PoC 进行动态验证。

核心理念是：**将安全约束从"人遵守的规则"转变为"系统自动执行的策略"。** 这与 Harness Engineering 的整体哲学完全一致——不信任概率性保证，只信任确定性约束。

## 十一、对未来的六个判断
基于以上全部分析，我对 Harness Engineering 的未来做出以下判断：

### 判断一：技术栈将收敛
Böckeler 预测 AI 会推动行业向**数量更少的技术栈和拓扑结构**收敛[^3]。选择技术的标准将不再仅仅是开发者偏好，而是**框架的"AI 友好程度"以及是否有成熟的 Harness 可用**。未来可能会出现针对常见应用拓扑的"Harness 模板"（类比今天的 Golden Path 服务模板），成为新项目的起点。

### 判断二：工程师角色不可逆转型
从"出租车司机"（关注怎么走这条路）转变为"城市规划师"（关注怎么把路修好）。核心工作的三要素变为：**拆解目标、构建脚手架、建立反馈回路**。OpenAI 的实践表明，AI 的自主度不是自然涌现的，而是工程师通过设计特定的结构和工具一点点"解锁"出来的。

### 判断三：严谨性重定位
工程的严谨性将从人工代码审查转移到**环境设计、控制系统构建和自动化护栏编写**上。纠错成本低于等待成本——这是 Harness 哲学的底层经济学原理。

### 判断四：新旧世界的鸿沟
对于从头构建的"后 AI 应用"，Harness 体系效果最好。对于遗留代码（旧应用），尝试"改造"一套 Harness 可能并不划算——旧的代码库往往充满了非标准化和混乱，就像在一个从未运行过静态分析工具的项目上突然开启它一样，会被警报淹没。

### 判断五："学徒缺口"将成为行业隐忧
如果初级工程师跳过手动开发直接用 AI 生成代码，虽然短期内效率高，但长期看会丧失建立架构直觉和判断代码质量的能力。**没有踩过坑的人，无法设计出管住 AI 的环境。** <u>Harness 的设计者必须是有深厚技术功底的资深工程师，这可能在未来几年形成人才断层。</u>

### 判断六：Harness 即护城河
Sam Altman 预言过"一个人的十亿美元公司"。在 Harness Engineering 的视角下，这个预言的前提条件变得清晰了：那个"一个人"不是一个更快的代码工人，而是一个**能设计出工业级 Harness 系统的能力架构师**——一个能让 16 个 Agent 并行运转两周而不崩溃的系统设计者。

---

## 十二、一句话总结
> **Harness Engineering 的本质，****是用确定性约束驯服概率性智能****。** 它不信任 Prompt 的祈祷，只信任 Lint 的阻断；不依赖模型的"常识"，只依赖架构的物理定律。在 2026 年，大模型本身的智力已经不是壁垒——谁能为 Agent 构建最严密的马具系统，谁就掌握了 Agentic Engineering 的制高点。
>

# 如何操作？
要让这个生物在庞大的代码库中不发疯、不遗忘、不破坏，你必须为它组装以下五个核心器官：

**宪法 (Constitution/静态知识) -> 潜意识技能 (Skills) -> 长时记忆 (Memory) -> 神经反射弧 (Hooks) -> 物理沙盒 (Harness)**。

让我们一层层切开，进行深度解剖。

---

### 🏛️ 第一层：宪法 (The Constitution) —— `CLAUDE.md`
**定位：绝对静态的底线法则（Always-On Context）**

在 2025 年，开发者喜欢把所有的东西（包括怎么连数据库、API 密钥在哪、之前的 Bug 怎么修的）全塞进 `CLAUDE.md`。这在 2026 年被证明是极其愚蠢的——它会导致严重的“Token 肥大症 (Token Bloat)”和缓存失效。

**2026 最新范式：极简宪法与 JIT 注入**

+ **剥离动态状态：**`CLAUDE.md` 中**绝对不能**出现任何动态进度（如“今天我们在修鉴权 Bug”）。它只定义跨越整个项目生命周期的绝对真理。例如：_“iOS 端强制使用 Swift 6 Strict Concurrency，禁用遗留的 completion handler；后端强制使用 Go 1.24+，并基于标准库的 _`_net/http_`_ 路由。”_
+ **全局加载：** 宪法是唯一一个在每个 Session 启动时，被强制锁定在 Context Window 最顶部的文档，以确保 100% 命中 Prompt Cache。
+ **宪法即代码：** 不要用模糊的自然语言写宪法。用 Markdown 的 Checklist 甚至 JSON Schema 定义。

---

### 🧠 第二层：潜意识与技能 (The Subconscious) —— `Skills` (SKILL.md)
**定位：按需加载的程序性记忆（On-Demand Procedural Knowledge）**

如果把所有的操作规范都写在宪法里，AI 会精神分裂。Claude Code 在 2026 年引入的最伟大特性之一就是 **Skills**。

+ **什么是 Skills？** 它是存放在 `.claude/skills/` 目录下的 Markdown 文件。比如你可以写一个 `TDD-Cycle.SKILL.md`，里面详细规定了“红-绿-重构”的每一步。
+ **与宪法的本质区别：按需唤醒。**`CLAUDE.md` 是 Agent 每秒钟都在思考的“重力法则”，而 Skill 是一本“字典”。当你在终端输入 `/tdd-cycle`，或者 Agent 判定当前任务需要写测试时，它才会把这部分 Token 动态加载到上下文中。
+ **实战收益：** 极大节省 Token。Agent 在写 UI 的时候，脑子里完全不需要装着“如何压测 Go 接口”的技能上下文。

---

### 💾 第三层：长时记忆 (The Hippocampus) —— `Memory` (OpenMemory /SuperMemory/ Memora)
**定位：跨越时空的防呆机制（Cross-Session Persistent Context）**

没有记忆的 Agent，就像一条只有 7 秒记忆的金鱼，每天都在踩昨天踩过的坑。

+ **2026 最新范式：从文件到 RAG 图谱**

放弃在根目录维护一个名为 `MEMORY.txt` 的屎山文件。全面接入 `memora` 或 `OpenMemory` 这样的本地向量/图谱数据库。

+ **核心机制：**
    - **自动墓碑化 (Automated Tombstoning)：** 业务逻辑是会变的。当 Agent 废弃了一个旧的 API 接口，Memory 系统不会物理删除它，而是打上 `[STALE]` 标签。下次 Agent 搜索时，系统会告诉它：“之前有这个接口，但上周被你干掉了，现在用新的。”
    - **语义直觉：** Agent 在面对莫名其妙的 Swift 编译报错时，不需要精准输入关键词，直接调用 MCP 的 `search_memories`，底层 Qdrant 向量库就能凭借语义，把两个月前解决同类报错的“心得”喂给它。

### ⚡ 第四层：神经反射弧 (The Nervous System) —— `Hooks`
**定位：绝对确定性的物理拦截（Deterministic Interception）**

大模型（LLM）是概率模型，它有可能会“发癫”。你不能在 Prompt 里哀求它：“求求你，提交代码前一定要跑一下 Lint。” —— **在 2026 年，不要用 Prompt 解决工程问题，用 Hooks。**

+ **双重 Hook 体系：**
    1. **Git Hooks (人类级拦截)：** 传统的 `pre-commit`，用来兜底，防止 Agent 生成的垃圾代码污染主分支。
    2. **Claude Code Hooks (Agent 级拦截 - 2026 杀手锏)：** 这是直接植入 AI 思考循环里的挂钩。
+ **2026 最佳实践：基于事件的 **`**PreToolUse**`** / **`**PostToolUse**`
    - 假设 Agent 决定调用 `edit_file` 工具修改一个极其核心的 Go 鉴权中间件。
    - `**PreToolUse**`** 拦截：** 钩子触发一个本地 Python 脚本，检测到修改敏感文件，直接返回 `Exit Code 2`，并在 stderr 中强行注入：“修改鉴权逻辑必须先生成一份 Architectural Decision Record (ADR)”。Agent 会被直接打断施法，乖乖去写文档。
    - `**PostToolUse**`** 拦截：** Agent 修改完 Swift 文件后，钩子强制静默运行 `swiftformat` 和 `swiftlint`。如果失败，直接把报错返回给 Agent 强迫重修。**这一切对人类是完全隐形的，你只看到最后完美的结果。**

### 🏟️ 第五层：终极裁判所与斗兽场 —— `The Harness`
**定位：脚手架工程，闭环自愈的物理沙盒 (The Runtime Arena)**

这是 2026 年 Agentic Engineering 的皇冠。**Harness 不是一个工具，它是一个“系统”。** 包含了我们前面聊过的所有的 CLI 和运行时环境。

如果说 MCP 是获取外部信息的触手，**那么 Harness 就是代码执行时的“生死簿”。**

#### 1. 从 MCP 迷信 回归 CLI 管道 (The Dual-Track CLI)
+ 2026 年的共识：**外环用 MCP，内环用 CLI。**
+ 当 Agent 需要在「磕线 Betaline」项目中验证一段代码逻辑时，绝不应该通过沉重的 MCP Server 去跑测试。Harness 应该被封装成一个纯粹的、本地的命令行工具（例如 `betaline-harness test --module=auth`）。
+ **Token 经济学：** LLM 天生就是 Linux 原住民，它精通组合各种 CLI 管道。这比让它去阅读庞大的 MCP Schema 要快 10 倍，成本低 10 倍。

#### 2. 流式 NDJSON 反馈 (Streaming Feedback Loop)
+ 传统的 Harness 跑完所有的测试，扔给 AI 一大段长达几千行的 Console Log。Agent 的注意力机制会完全涣散。
+ **2026 的 Harness 必须实现 TTY 嗅探：** 当检测到运行者是 Agent 时，立刻关闭所有进度条动画，切换为 `NDJSON`（换行符分隔的 JSON）流式输出。

#### 3. 行动建议注入 (Actionable Hints)
这是 Harness 区分“传统 CI”和“Agent-Native 沙盒”的核心标志。

当 Agent 跑挂了测试，Harness 抛出的 JSON 绝对不能只是：`{"error": "Index out of range"}`。

它必须具备对 LLM 的同理心，抛出如下结构：

JSON

```plain
{
  "status": "FAILED",
  "stage": "Boundary Validation",
  "file": "pkg/route_calculator/grade.go",
  "line": 128,
  "error": "panic: assignment to entry in nil map",
  "agent_action_hint": "CRITICAL: In Go, you cannot assign to a nil map. Please initialize the map using 'make(map[string]int)' before line 128. Re-read CLAUDE.md section on Go memory safety."
}
```

**这种带有 **`**agent_action_hint**`** 的强制反馈回路，就是将“开环的盲写”转变为“闭环的自动修正 (Autonomous Healing)”的根本原因。**

---

### 🎯 结语：从“工具”到“生态系统”
回顾这五层架构，你会发现 2026 年的最佳范式，本质上是对 Agent 进行了**“能力降权”**与**“环境升维”**。

我们不再指望用一句充满魔力的魔法提示词（Prompt）让大模型一次性写出完美的系统。相反，我们承认大模型是发散的、会遗忘的、容易犯错的。

+ 我们用 `**CLAUDE.md**` 和 `**Skills**` 控制它的注意力；
+ 我们用 `**Memory**` 弥补它的短期遗忘；
+ 我们用 `**Hooks**` 剥夺它犯低级错误的权利；
+ 最后，我们将它扔进由 **NDJSON CLI** 构筑的 `**Harness**` 斗兽场中，让机器用机器能看懂的语言去鞭策机器。

作为一个人类架构师，你的工作已经从“逐行敲击代码”变成了“设计、维护并不断强化这个 Harness 体系”。当你把这个系统搭建完毕，你就可以放心地让几十个 Agent 并发运行在你的代码库中，它们会日夜不休地自我修正、自我迭代。
